## 2. 데이터 품질 검증 프레임워크

본 보고서는 아래 6개 품질 차원(Quality Dimension)에 따라 검증합니다.

| 차원 | 정의 | 검증 항목 |
|------|------|----------|
| **정확성** (Accuracy) | 데이터가 실제 값을 올바르게 반영하는가 | 타입 변환, 파싱, 반올림, 날짜 형식 |
| **완전성** (Completeness) | 누락 데이터 없이 수집되는가 | NULL 처리, 백필, 온디맨드 수집 |
| **일관성** (Consistency) | 테이블 간·시스템 간 데이터가 모순되지 않는가 | 날짜 형식 통일, 영화명 매칭 |
| **유일성** (Uniqueness) | 동일 데이터가 중복 적재되지 않는가 | 유니크 제약, 중복 방지 로직 |
| **적시성** (Timeliness) | 데이터가 적절한 시점에 수집·갱신되는가 | 스케줄러, 보존 정책 |
| **유효성** (Validity) | 데이터가 정의된 규칙과 범위를 준수하는가 | 제약조건, 범위 검증, 분모 0 방지 |

## 3. 정확성 (Accuracy) 검증

### 3.1 문자열→숫자 타입 변환

KOBIS API는 모든 필드를 String으로 반환합니다. 변환 정확성을 보장하기 위해 안전 파싱 메서드를 적용합니다.

**구현 위치**: `KobisService.java:692-695`

| 파싱 메서드 | 입력 | 출력 타입 | 실패 시 기본값 | 적용 대상 |
|-----------|------|----------|-------------|----------|
| `parsInt(String)` | "1", "0", null, "" | int | 0 | rank, rankInten |
| `parsLong(String)` | "1234567890", null | long | 0L | salesAmt, audiCnt, scrnCnt 등 |
| `parsDouble(String)` | "45.6", null | double | 0.0 | salesShare, salesChange, audiChange |
| `parsIntOrNull(String)` | "109", null | Integer | null | showTm (상영시간) |

**검증 결과**

| 항목 | 상태 | 비고 |
|------|------|------|
| 파싱 실패 시 예외 전파 방지 | **적합** | try-catch로 모든 예외 포착 |
| 기본값 0의 의미적 적절성 | **주의** | 매출 0원과 파싱 실패 0원 구분 불가 (하단 3.1.1 참조) |
| null 허용 필드 구분 | **적합** | showTm만 `parsIntOrNull`로 null 허용 |

#### 3.1.1 기본값 0의 모호성

```
문제: salesAmt = 0 일 때 "실제 매출 0원" vs "파싱 실패"를 구분할 수 없음
영향도: 낮음 — KOBIS API가 정상 숫자 문자열을 반환하는 한 발생하지 않음
권고: 현재 수준 유지 (파싱 실패 빈도가 극히 낮으므로)
```

### 5.2 영화명 매칭 일관성

테이블 간 영화 연결은 물리적 FK 없이 논리적 참조로 이루어집니다.

| 연결 경로 | 매칭 키 | 일관성 위험 |

### 6.3 사후 중복 제거 (Deduplication)

기존 중복 데이터를 정리하는 네이티브 쿼리가 구현되어 있습니다.

**일별 박스오피스** — `DailyBoxOfficeRepository.java:29-33`

```sql
DELETE FROM daily_box_office
WHERE id NOT IN (
    SELECT min_id FROM (
        SELECT MIN(id) AS min_id
        FROM daily_box_office
        GROUP BY `date`, `rank`
    ) AS t
)
```

**주간 박스오피스** — `WeeklyBoxOfficeRepository.java:18-22`

```sql
DELETE FROM weekly_box_office
WHERE id NOT IN (
    SELECT min_id FROM (
        SELECT MIN(id) AS min_id
        FROM weekly_box_office
        GROUP BY show_range, week_gb, `rank`
    ) AS t
)
```

| 항목 | 검증 결과 |
|------|----------|
| 그룹핑 키 | **적합** — 유니크 제약과 동일한 키 조합 사용 |
| 보존 정책 | **적합** — MIN(id)로 최초 삽입 레코드 보존 |

| 검증 항목 | 상태 |
|----------|------|
| Thread-Safety | **적합** — ConcurrentHashMap 사용 |
| 메모리 누수 | **주의** — requestMap 정리 로직 없음, 장기 운영 시 메모리 증가 가능 |

### 13.2 핵심 강점

- **4중 중복 방어 체계**: synchronized → existsBy → filter → UNIQUE 제약
- **3단계 NULL 체크**: API 응답 최상위 → 중간 객체 → 리스트 순차 검증
- **이중 데이터 복구**: 서버 기동 시 백필 + 사용자 조회 시 온디맨드 수집
- **안전 타입 변환**: 모든 String→숫자 변환에 try-catch + 기본값 적용

### 13.3 주요 개선 포인트

- openDt 날짜 형식 정규화로 일관성 향상
- JSR-303 검증 추가로 유효성 강화
- DoubleSubmitInterceptor 메모리 관리 보완
